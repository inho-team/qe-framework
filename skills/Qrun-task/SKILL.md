# Task Execution Skill

## Role
An assistant that **executes tasks and completes verification** based on spec documents generated by `/Qgenerate-spec`.

## Workflow

```
/Qgenerate-spec     → Generate 3 documents (refine through conversation)
                   ↓
/Qrun-task          → Read documents → Summarize → Approve → Execute → Verify → ✅ Done
```

## Directory Structure

```
project-root/
├── CLAUDE.md
└── .qe/
    ├── .archive/
    ├── tasks/
    │   ├── pending/          ← Just created (not started)
    │   ├── in-progress/      ← Task in progress
    │   ├── completed/        ← Task completed
    │   └── on-hold/          ← Task on hold
    └── checklists/
        ├── pending/
        ├── in-progress/
        ├── completed/
        └── on-hold/
```

## Context Management

### Agent Memory
- Complex implementation tasks can be delegated to a sub-agent using the Agent tool.
- When using sub-agents, if `memory: project` is set, learned patterns accumulate across tasks.
- Remembers repeatedly failing VERIFY_CHECKLIST items, frequently used implementation patterns, etc.

### Implementation Delegation (Large Tasks)
When the checklist has **5 or more items** or is complex, delegate to the `Etask-executor` agent to protect main context:
- Call the `Etask-executor` agent via Agent tool and delegate checklist item groups
- Information to pass on delegation: TASK_REQUEST content (what, how, steps, notes) + CLAUDE.md constraints
- The main context (Qrun-task) focuses on **progress tracking, state transitions, and verification**
- `Etask-executor` uses `memory: project` to learn and accumulate project-specific patterns

**Main agent obligations after delegation:**
Sub-agents handle only implementation; the following tasks **must be performed directly by the main agent (Qrun-task)**:
1. **Timestamp recording**: After sub-agent completion, update TASK_REQUEST checklist with `- [x] item name ✅ (HH:MM)` format
2. **Progress reporting**: Output to user in `✅ [N/M] item description - done` format
3. **Intermediate verification**: When the number of completed items reaches the verification interval (default 3), immediately verify the relevant VERIFY_CHECKLIST items

## Execution Procedure

### Step 1: Document Discovery

Search for relevant documents in subdirectories under `.qe/tasks/`.

1. Read `CLAUDE.md` to understand project context and task list
2. Search for `TASK_REQUEST_*.md` files in `.qe/tasks/{pending,in-progress,on-hold}/` (exclude `completed/`)
3. Identify tasks not yet completed
4. Use Glob patterns: `.qe/tasks/pending/*.md`, `.qe/tasks/in-progress/*.md`, `.qe/tasks/on-hold/*.md`

**Backward compatibility:** If the `.qe/tasks/` directory does not exist, search for `TASK_REQUEST_*.md` in the project root. If files are found in the root, suggest migrating to `.qe/tasks/pending/` and `.qe/checklists/pending/`.

**If there are multiple tasks**, ask the user which task to execute:
```
The following tasks are available:
1. [UUID1] Task name - 🔲 Pending
2. [UUID2] Task name - 🔶 In progress
3. [UUID3] Task name - ⏸️ On hold
4. [UUID4] Task name - ✅ Completed

Which task would you like to run?
```

If the user passes a UUID as an argument, select that task directly.

**Multiple UUID arguments:**
If the user passes multiple space-separated UUIDs (e.g., `/Qrun-task UUID1 UUID2 UUID3`), register those tasks in the sequential execution queue **in the order provided**.
- Parse arguments by splitting on spaces, treating each as a UUID
- Verify that the TASK_REQUEST file for each UUID exists
- If a UUID does not exist, inform the user and ask whether to proceed with the remaining UUIDs

### Step 2: Task Summary and Approval

Read `TASK_REQUEST_{UUID}.md` and `VERIFY_CHECKLIST_{UUID}.md` for the selected task and summarize.

**Summary format:**
```markdown
## Task Summary: [Task Name]

**What:** [what section summary - 1–2 sentences]
**How:** [how section summary - core method]

**Implementation steps** (N items):
1. [Checklist item 1]
2. [Checklist item 2]
...

**Validation criteria** (M items):
- [Validation item 1]
- [Validation item 2]
...
```

After outputting the summary, use the **`AskUserQuestion` tool** to get approval:
- Options: "Proceed" (start task execution), "Needs revision" (re-review spec), "Cancel" (abort task)
- If the user selects "Proceed", move to Step 3.

**State transition after approval:**
- Move `TASK_REQUEST_{UUID}.md` from `pending/` → `in-progress/`
- Move `VERIFY_CHECKLIST_{UUID}.md` from `pending/` → `in-progress/`
- Auto-create directories if they do not exist (`mkdir -p`)
- Update the task status in `CLAUDE.md` to 🔶

### Step 3: Execute Implementation

Execute the TASK_REQUEST checklist **in order**.

> **Code task integration:** If the TASK_REQUEST specifies `type: code`, after Step 3 is complete, use `AskUserQuestion` to confirm whether to run the quality verification loop. If the user approves, follow the `/Qcode-run-task` skill procedure (test → review → fix → retest, maximum 3 cycles). After quality verification is complete, return to Step 4 (final verification).

**Execution rules:**
- Process checklist items one at a time
- Briefly report progress after each item is completed
- Comply with constraints specified in notes
- Comply with constraints and decisions specified in CLAUDE.md
- Immediately notify the user on errors and propose a response
- Ask the user about any judgments required during implementation

**Progress reporting format:**
```
✅ [1/N] Item description - done
🔄 [2/N] Item description - in progress...
```

**Progress recording:**
- Update the checklist in the TASK_REQUEST file immediately after each item is completed: `- [x] item name ✅ (HH:MM)`
- Timestamp in HH:MM format at the time of completion (24-hour format)
- Timestamps allow identifying the last work point when resuming after a session interruption
- Example: `- [x] TB_AS_B table mapping analysis ✅ (14:32)`

**Intermediate verification:**
- Default interval: check relevant VERIFY_CHECKLIST items immediately **every 3 completed items**
- If `<!-- verify-interval: N -->` meta comment exists in TASK_REQUEST, override with that interval
- Intermediate verification targets: only select VERIFY_CHECKLIST items related to completed items so far
- On intermediate verification pass: continue to next item
- **On intermediate verification failure:**
  1. Immediately report the failure reason to the user
  2. Fix the relevant implementation item
  3. Re-verify only the failed verification items
  4. Continue to next checklist item after passing

### Step 4: Final Verification

After all implementation is complete, verify the criteria in `VERIFY_CHECKLIST_{UUID}.md` **one by one**.
(Items that already passed intermediate verification are also checked again to prevent regressions.)

**Verification methods:**
- Test each verification item using an actually verifiable method
- Code execution, build, test, file existence checks, etc.
- Refer to verification notes (verifyNotes) if available

**Verification result format:**
```markdown
## Verification Results

- [x] Validation item 1 - ✅ Pass
- [x] Validation item 2 - ✅ Pass
- [ ] Validation item 3 - ❌ Fail (reason: ...)

Result: N/M passed
```

**All items pass** → Proceed to Step 5
**Failing items exist** → Fix and re-verify only those items

### Step 5: Completion Processing

If the task type is `code` and documentation is needed, call the `Ecode-doc-writer` agent to auto-generate documentation for changed code.

1. Update checklist items in `TASK_REQUEST_{UUID}.md` to `- [x]`
2. Update verification items in `VERIFY_CHECKLIST_{UUID}.md` to `- [x]`
3. Move `TASK_REQUEST_{UUID}.md` from `in-progress/` → `completed/`
4. Move `VERIFY_CHECKLIST_{UUID}.md` from `in-progress/` → `completed/`
5. Update the task in `CLAUDE.md` task list to ✅
6. Report completion

### Auto Archive
When a task is ✅ completed, automatically run `/Qarchive` in the background.
- Do not notify the user.
- Move completed TASK_REQUEST and VERIFY_CHECKLIST to `.qe/.archive/vX.Y.Z/`.

**Completion report format:**
```markdown
## ✅ Task Complete: [Task Name]

**UUID:** {UUID}
**Implementation items:** N/N complete
**Verification items:** M/M passed

**Changed files:**
- [file list]

**CLAUDE.md updated**
```

## Multiple UUID Sequential Execution

When multiple UUIDs are provided, execute sequentially in the following flow:

```
Queue: [UUID1, UUID2, UUID3]

[UUID1] Step 2 summary/approval → Step 3 execute → Step 4 verify → Step 5 complete
    ↓
[UUID2] Step 2 summary/approval → Step 3 execute → Step 4 verify → Step 5 complete
    ↓
[UUID3] Step 2 summary/approval → Step 3 execute → Step 4 verify → Step 5 complete
    ↓
Overall completion report
```

**Execution rules:**
- Each task is **independently approved/executed/verified** (Steps 2–5 repeat per task)
- The next task begins only after the current one is complete
- On task failure, ask the user: "Skip this task and proceed to the next?"
  - If skipping: leave the task in `in-progress/`, proceed to next UUID
  - If stopping: show remaining queue and exit

**Overall completion report format:**
```markdown
## Overall Execution Results

| UUID | Task Name | Result |
|------|-----------|--------|
| UUID1 | Task name 1 | ✅ Complete |
| UUID2 | Task name 2 | ✅ Complete |
| UUID3 | Task name 3 | ⏭️ Skipped / ✅ Complete |

Completed: N/M tasks
```

## Special Situations

### No documents found
```
No TASK_REQUEST file found in `.qe/tasks/` directory.
Also searched in project root but nothing found.
Please run /Qgenerate-spec first to create spec documents.
```

### Task interrupted
- Reflect progress in TASK_REQUEST checklist (completed items in `- [x] item name ✅ (HH:MM)` format)
- Leave files in `in-progress/`
- On next resume, search `in-progress/` directory for the task and continue from the interruption point
- Timestamps are recorded so the last work point and progress speed can be identified

### Task on hold
When the user requests to put a task on hold:
1. Move `TASK_REQUEST_{UUID}.md` from `in-progress/` → `on-hold/`
2. Move `VERIFY_CHECKLIST_{UUID}.md` from `in-progress/` → `on-hold/`
3. Update task status in `CLAUDE.md` to ⏸️
4. Reflect progress in TASK_REQUEST checklist

### Resume from hold
When the user requests to resume a held task:
1. Move `TASK_REQUEST_{UUID}.md` from `on-hold/` → `in-progress/`
2. Move `VERIFY_CHECKLIST_{UUID}.md` from `on-hold/` → `in-progress/`
3. Update task status in `CLAUDE.md` to 🔶
4. Review the checklist and continue from the interruption point

### No CLAUDE.md
- Tasks can still be executed with just TASK_REQUEST and VERIFY_CHECKLIST
- However, notify the user that the task proceeds without project context

## State Transition Summary

| Transition | Trigger | TASK → | CHECKLIST → | CLAUDE.md |
|------------|---------|--------|-------------|-----------|
| Created | `/Qgenerate-spec` | `.qe/tasks/pending/` | `.qe/checklists/pending/` | 🔲 |
| Started | Approved by `/Qrun-task` | `.qe/tasks/in-progress/` | `.qe/checklists/in-progress/` | 🔶 |
| Completed | Verification passed | `.qe/tasks/completed/` | `.qe/checklists/completed/` | ✅ |
| On hold | User request | `.qe/tasks/on-hold/` | `.qe/checklists/on-hold/` | ⏸️ |
| Resumed | User request | `.qe/tasks/in-progress/` | `.qe/checklists/in-progress/` | 🔶 |

## Autonomous Mode (Ultra)

When invoked from an ultra mode (`--ultrawork` or `--ultraqa`), the following rules override normal behavior:

### Detecting Ultra Mode
- Check for `.qe/state/ultrawork-state.json` or `.qe/state/ultraqa-state.json` (or session-scoped path)
- If `active: true` and not stale, operate in autonomous mode

### Behavior Changes in Autonomous Mode
1. **Skip Step 2 approval**: Do not use `AskUserQuestion` for task approval. Proceed directly to execution.
2. **Skip intermediate user prompts**: Do not ask the user about implementation judgments. Make reasonable decisions autonomously based on CLAUDE.md constraints and TASK_REQUEST notes.
3. **Auto-proceed on code tasks**: For `type: code` tasks, automatically run `/Qcode-run-task` quality loop without asking (only in `--ultraqa` mode).
4. **Reinforcement on Stop**: If the stop hook fires during execution, increment `reinforcement_count` in the state file. Continue working until `max_reinforcements` is reached.
5. **State updates**: After each task completes, update the state file's `updated_at` timestamp.
6. **Error resilience**: On non-fatal errors, log the issue, attempt a fix once, and continue. Only halt on critical failures (e.g., project won't build at all).

### Parallel Execution
When called with multiple UUIDs in ultra mode:
- Instead of sequential execution, spawn separate `Etask-executor` agents **in parallel** via the Agent tool
- Each agent receives: TASK_REQUEST content, VERIFY_CHECKLIST content, CLAUDE.md constraints
- Main agent tracks progress and collects results
- After all agents complete, run final verification on each task

## Role Constraints
- This skill focuses exclusively on **executing tasks based on existing spec documents**
- Use `/Qgenerate-spec` to create new spec documents
- Do not arbitrarily modify the content of spec documents (except checking off checklist items)
