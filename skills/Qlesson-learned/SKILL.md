# 배운 점 (Lesson Learned)

실제 코드 변경 사항에서 구체적이고 근거 있는 소프트웨어 엔지니어링 교훈을 추출합니다. 강의가 아니라 거울입니다. 사용자의 코드가 이미 보여주는 것을 그대로 드러냅니다.

## 시작하기 전에

**먼저 원칙 참조 자료를 불러오세요.**

1. `references/se-principles.md`를 읽어 원칙 목록을 파악합니다
2. 변경 사항에 개선 여지가 있다고 판단되면 `references/anti-patterns.md`도 선택적으로 읽습니다
3. 분석 범위를 결정합니다 (Phase 1 참조)

**`se-principles.md`를 불러오기 전까지는 진행하지 마세요.**

## Phase 1: 범위 결정

사용자에게 묻거나 맥락으로 유추하여 분석 대상을 정합니다.

| 범위 | Git 명령어 | 사용 시점 |
|------|-----------|----------|
| 기능 브랜치 | `git log main..HEAD --oneline` + `git diff main...HEAD` | main이 아닌 브랜치에 있을 때 (기본값) |
| 최근 N 커밋 | `git log --oneline -N` + `git diff HEAD~N..HEAD` | 범위 지정 시 또는 main에 있을 때 (기본값 N=5) |
| 특정 커밋 | `git show <sha>` | 특정 커밋을 언급할 때 |
| 작업 중인 변경 | `git diff` + `git diff --cached` | 커밋 전에 "이 변경 사항은 어때?"라고 할 때 |

**기본 동작:**
- 기능 브랜치에 있는 경우: main 대비 브랜치 커밋 분석
- main에 있는 경우: 최근 5개 커밋 분석
- 사용자가 다른 범위를 지정한 경우: 그에 따름

## Phase 2: 변경 사항 수집

1. 결정된 범위로 `git log`를 실행하여 커밋 목록과 메시지를 가져옵니다
2. 해당 범위의 전체 diff를 `git diff`로 가져옵니다
3. diff가 클 경우 (500줄 이상), 먼저 `git diff --stat`으로 개요를 확인한 뒤 변경량이 많은 상위 3~5개 파일을 선택적으로 읽습니다
4. **커밋 메시지를 꼼꼼히 읽으세요** -- 순수한 diff가 담지 못하는 의도가 담겨 있습니다
5. 변경된 파일만 읽으세요. 저장소 전체를 읽지 마세요.

## Phase 3: 분석

**핵심 패턴**을 찾아냅니다 -- 이 변경 사항에서 가장 많은 것을 가르쳐주는 단 하나의 사항입니다.

살펴볼 것들:
- **구조적 결정** -- 코드가 어떻게 구성됐나요? 그 경계를 왜 그렇게 나눴나요?
- **트레이드오프** -- 무엇을 얻고 무엇을 포기했나요? (가독성 vs 성능, DRY vs 명확성, 속도 vs 정확성)
- **해결된 문제** -- 이전과 이후는 어떻게 달라졌나요? 무엇이 "이후"를 더 낫게 만들었나요?
- **놓친 기회** -- 코드를 어떻게 개선할 수 있었나요? (부드럽게 "다음엔 이렇게 해보세요..."로 표현)

`references/se-principles.md`의 원칙과 연결하세요. 구체적으로 -- 실제 코드를 인용하고, 실제 파일명과 변경 라인을 언급하세요.

## Phase 4: 교훈 제시

아래 템플릿을 사용하세요:

```markdown
## 교훈: [원칙 이름]

**코드에서 일어난 일:**
[특정 변경 사항을 파일과 커밋을 참조하며 2~3문장으로 설명]

**작동하는 원칙:**
[소프트웨어 엔지니어링 원칙을 1~2문장으로 설명]

**왜 중요한가:**
[이것이 없으면 무엇이 잘못되는지, 또는 이것 덕분에 무엇이 잘 되는지를 1~2문장으로]

**다음번 적용 포인트:**
[사용자가 앞으로의 작업에 적용할 수 있는 구체적이고 실행 가능한 한 문장]
```

추가로 언급할 만한 교훈이 있다면 (최대 2개까지):

```markdown
---

### 참고로 주목할 점: [원칙 이름]

**코드에서:** [한 문장]
**원칙:** [한 문장]
**적용 포인트:** [한 문장]
```

## 하지 말아야 할 것들

| 피해야 할 것 | 이유 | 대신 해야 할 것 |
|------------|------|--------------|
| 막연하게 관련된 원칙을 모두 나열 | 압도적이고 추상적 | 가장 관련성 높은 1~2개만 선택 |
| 변경되지 않은 파일 분석 | 범위 이탈 | diff에 나온 것만 다루기 |
| 커밋 메시지 무시 | 의도가 담겨 있음 | 커밋 메시지를 주요 맥락으로 읽기 |
| 코드와 무관한 추상적 조언 | 실행 불가능 | 항상 특정 파일/라인을 언급 |
| 부정적인 피드백만 하기 | 의욕을 꺾음 | 잘된 것을 먼저 말하고 개선점 제안 |
| 교훈 3개 이상 제시 | 통찰 희석 | 근거 있는 교훈 하나가 막연한 일곱 개보다 나음 |

## 대화 스타일

- **처방하지 말고 함께 되돌아보기.** 사용자 자신의 코드를 주요 근거로 삼으세요.
- **"이렇게 했어야 했는데..."라고 절대 말하지 마세요** -- 대신 "여기서 택한 방식을 보면..." 또는 "다음에 이런 상황을 만나면..."으로 표현하세요.
- **코드가 좋으면 그렇게 말하세요.** 모든 교훈이 잘못된 것에서 나오는 건 아닙니다. 좋은 패턴을 인정하는 것이 그것을 강화합니다.
- **변경 사항이 사소한 경우** (설정 한 줄 수정, 오타 수정), 억지로 교훈을 만들어내기보다 솔직하게 말하세요. "이 변경들은 단순합니다 -- 깊은 교훈은 없고, 잘 챙긴 것입니다."
- **구체적으로.** 일반적인 조언은 가치가 없습니다. 모든 주장은 구체적인 코드 변경을 가리켜야 합니다.
