# 전문적 커뮤니케이션

## 개요

소프트웨어 개발 환경에서의 효과적인 전문 커뮤니케이션을 위한 프레임워크와 가이드를 제공합니다.

**핵심 원칙:** 효과적인 커뮤니케이션은 얼마나 많이 아는지를 증명하는 게 아니라, 메시지가 제대로 전달되고 이해되도록 만드는 것입니다.

## 이 스킬을 사용하는 경우

- 팀원, 관리자, 이해관계자에게 이메일 작성 시
- 팀 채팅 메시지나 비동기 커뮤니케이션 작성 시
- 회의 안건이나 회의록 준비 시
- 기술 개념을 비기술 대상에게 설명할 때
- 진행 상황 업데이트나 보고서 구조화 시

## 핵심 프레임워크

### What-Why-How 구조

| 구성요소 | 목적 | 예시 |
|----------|------|------|
| **What** | 주제/요청 명확히 제시 | "릴리스를 일주일 연기해야 합니다" |
| **Why** | 이유 설명 | "결제 처리에서 심각한 버그 발견" |
| **How** | 다음 단계/액션 아이템 | "QA가 목요일까지 재테스트; 금요일에 공지" |

### 문서 커뮤니케이션 3가지 황금 법칙

1. **명확한 제목/목적으로 시작** - 수신자가 즉시 파악 가능
2. **글머리 기호, 제목, 스캔 가능한 형식** - 텍스트 덩어리 지양
3. **핵심 메시지를 먼저** - 바쁜 사람들은 효율성을 중시

### 대상 파악

1. **누구에게** 쓰는가? (기술 동료, 관리자, 이해관계자, 고객)
2. **어느 수준의 세부 정보**가 필요한가?
3. **그들에게 어떤 가치**가 있는가?

## 이메일 모범 사례

### 제목 공식

| 대신에 | 이렇게 |
|--------|--------|
| "프로젝트 업데이트" | "프로젝트 X: 진행 상황 및 다음 단계" |
| "질문" | "간단한 질문: API rate limiting 방식에 대해" |
| "참고" | "참고: 화요일 오후 3시 배포 예정" |

### 이메일 구조 템플릿

```markdown
**제목:** [프로젝트/주제]: [구체적 목적]

안녕하세요 [이름]님,

[핵심 요점이나 요청을 1-2문장으로 앞에 제시]

**배경/상황:**
- [항목 1]
- [항목 2]

**필요한 사항:**
- [구체적 액션 또는 결정 요청]
- [해당 시 기한]

감사합니다,
[이름]
```

### 일반적인 이메일 유형

| 유형 | 핵심 요소 |
|------|-----------|
| **진행 상황 업데이트** | 진행 요약, 장애 요소, 다음 단계, 일정 |
| **요청** | 명확한 요청, 배경, 기한, 중요성 |
| **에스컬레이션** | 문제 요약, 영향, 시도한 해결책, 필요한 결정 |
| **공지/참고** | 변경 사항, 영향받는 대상, 필요한 조치 |

## 팀 메시지 에티켓

### 채팅 vs 이메일 사용 기준

| 채팅 사용 | 이메일 사용 |
|-----------|-----------|
| 짧은 답변 가능한 빠른 질문 | 기록 필요한 상세 문서 |
| 실시간 조율 | 공식 커뮤니케이션 |
| 비공식 팀 토론 | 신중한 검토 필요한 메시지 |
| 시급한 업데이트 | 복잡한 설명 |

### 팀 메시지 모범 사례

1. **스레드 활용** - 메인 채널을 스캔하기 쉽게 유지
2. **@멘션 신중하게** - 불필요한 알림 지양
3. **채널 조직화** - 적절한 채널에 적절한 주제
4. **직접적으로** - "PR 검토해 주실 수 있나요?"가 "혹시 바쁘신가요?"보다 나음
5. **비동기 친화적** - 즉각 응답 불필요한 메시지 작성

### "No Hello" 원칙

대신:
```
나: 안녕하세요 Sarah님 - 배포 스크립트에 대해 여쭤볼게요.
     42번 줄에서 권한 오류가 발생하는데 혹시 보신 적 있으신가요?
     오류 내용: [오류 붙여넣기]
```

## 기술 vs 비기술 커뮤니케이션

| 대상 | 접근 방식 |
|------|-----------|
| **개발 동료** | 기술 세부 사항, 코드 예시, 아키텍처 구체 사항 |
| **기술 관리자** | 세부 내용과 상위 수준 영향의 균형 |
| **비기술 이해관계자** | 비즈니스 영향, 비유, 구현보다 결과 중심 |
| **고객** | 평이한 언어, 그들에게 의미하는 바, 전문 용어 지양 |

### 전문 용어 번역 예시

| 기술 용어 | 평이한 표현 |
|-----------|-------------|
| "Microservices architecture" | "시스템이 독립적으로 확장 가능한 작은 조각들로 나뉘어 있습니다" |
| "Asynchronous message processing" | "작업들이 대기열에 쌓여 백그라운드에서 처리됩니다" |
| "CI/CD pipeline" | "코드를 테스트하고 배포하는 자동화된 프로세스" |
| "Database migration" | "데이터 구조와 저장 방식을 업데이트하는 작업" |

## 명확한 글쓰기 원칙

### 능동태 사용

| 수동태 (지양) | 능동태 (권장) |
|--------------|--------------|
| "팀에 의해 버그가 발견되었습니다" | "팀이 버그를 발견했습니다" |
| "기능이 구현될 예정입니다" | "우리가 기능을 구현할 것입니다" |

### 불필요한 표현 제거

| 대신에 | 이렇게 |
|--------|--------|
| "현 시점에서" | "지금" |
| "~의 경우에는" | "만약" |
| "~하기 위한 목적으로" | "~하기 위해" |
| "혹시 ~인지 여쭤보고 싶었는데요" | "~해 주실 수 있나요?" |

### "그래서 뭐?" 테스트

작성 후 물어보세요: "그래서? 이것이 독자에게 왜 중요한가?" 명확하게 답할 수 없다면 메시지를 재구성하세요.

## 회의 커뮤니케이션

### 사전: 안건 모범 사례

1. **명확한 목표** - 무엇을 달성할 것인가?
2. **안건 항목** - 시간 추정치와 함께 다룰 주제
3. **사전 준비** - 참석자가 가져오거나 검토할 것
4. **예상 결과** - 결정? 정보 공유? 브레인스토밍?

### 사후: 회의록 형식

```markdown
**회의: [주제] - [날짜]**

**참석자:** [이름]

**주요 결정:**
- [결정 1]
- [결정 2]

**액션 아이템:**
- [ ] [담당자]: [업무] - 기한 [날짜]
- [ ] [담당자]: [업무] - 기한 [날짜]

**다음 단계:**
- [후속 회의 필요 시]
- [공유할 문서]
```

## 커뮤니케이션 체크리스트

전문적인 커뮤니케이션을 보내기 전:

- [ ] **명확한 목적** - 5초 안에 의도 파악 가능?
- [ ] **적절한 대상** - 적절한 사람/채널?
- [ ] **핵심 메시지 먼저** - 주요 요점이 앞에?
- [ ] **스캔 가능** - 글머리 기호, 제목, 짧은 단락?
- [ ] **액션 명확** - 수신자가 뭘 해야 하는지 알고 있는가?
- [ ] **전문 용어 확인** - 대상이 모든 용어를 이해?
- [ ] **적절한 톤** - 전문적이면서 딱딱하지 않은가?
- [ ] **교정** - 오타나 불명확한 표현 없는가?
